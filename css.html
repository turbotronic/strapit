Mobile first

Instead of adding on optional mobile styles, Strapit is mobile first. Mobile first styles can be found throughout the entire library instead of in separate files.

To ensure proper rendering and touch zooming, add the viewport meta tag to your <head>.

{% highlight html %} {% endhighlight %}

You can disable zooming capabilities on mobile devices by adding user-scalable=no to the viewport meta tag. This disables zooming, meaning users are only able to scroll, and results in your site feeling a bit more like a native application. Overall we don't recommend this on every site, so use caution!

Responsive images

Images in Strapit are responsive-friendly by default. If you're using Google Maps, you might run into some display problems due to the use of max-width on images. Also, there may be a problem with the use of * { box-sizing: border-box; }. The following snippet should avoid all those problems.

{% highlight css %} /* Fix Google Maps canvas * * Wrap your Google Maps embed in a `.google-map-canvas` to reset Strapit's * global `box-sizing` changes. You may optionally need to reset the `max-width` * on images in case you've applied that anywhere else. (That shouldn't be as * necessary with Strapit though as that behavior is relegated to the * `.img-responsive` class.) */ .google-map-canvas, .google-map-canvas * { .box-sizing(content-box); max-width: none !important; img { display: inline; } } {% endhighlight %}

For other cases where max-width is not desired on an image, or any other element for that matter, add the .no-max class.

Typography and links

Strapit sets basic global show-grid, typography, and link styles. Specifically, we:

  • Remove margin on the body
  • Add position: relative on the body for use in off-canvas layouts
  • Set background-color: #fff; on the body
  • Use the @font-family-base, @font-size-base, and @line-height-base attributes as our typographic base
  • Set the global link color via @link-color and apply link underlines only on :hover

These styles can be found within scaffolding.less.

Normalize

For improved cross-browser rendering, we use Normalize, a project by Nicolas Gallagher and Jonathan Neal. However, rather than one large file, Normailize is split up within the framework where the appropriate styles are grouped the LESS file that they impact. For example, you'll find Normailze's fonr styles within forms.less.

The advantage of this approach is that you use only the styles you need, so if you did not need forms.less, you would not get hte reset styles, either. The disadvantage is that it is harder to maintain when Normalize updates. For that reason, each Normalize style is commented with a version number e.g.: //(nm v2.1.3).

Grid system

Strapit includes a responsive, mobile-first fluid grid system that appropriately scales up to 12 columns as the device or viewport size increases. It includes predefined classes for easy layout options, as well as powerful mixins for generating more semantic layouts.

But first, a word on the evolving philosophies behind responsive design

You may notice that we have gone with the Foundation grid (with some modifications, such as the addition of Foundation's currently "experimental" .md class) over Bootstrap 3.0. The reason for this is we feel the Foundation grid is simpler and subsequently easier to modify.

Bootstrap provides for four distinct sizes that resolve to phone, tablets, desktops and large desktops in practice. We don't think it is useful to think of breakpoints in this way.

Instead, breakpoints should apply when the design needs to change to accommodate the viewport within which it is contained. So the breakpoints could be as few as one or as many as 20, the important consideration is that the site is functional and viewable at all sizes from 320px and up.

Obviously this is tough to accommodate for in a framework. So, we feel the winning grid is the one that has the least code to change, which, as of today, is Foundation's. That said, we like Bootstrap's class names and conventions better, so we used those.

Media queries

We use the following media queries to create the key breakpoints in our grid system. These are all optional and can be turned on and off as needed

{% highlight html %} /* Extra-small devices (phones up to 479px) */ @media screen and (min-width: @site-min-width) { ... } /* Small devices (large phonesn and small tables up to 767px) */ @media screen and (min-width: @breakpoint-sm) { ... } /* Medium devices (tablets to small desktops up to 959px) */ @media screen and (min-width: @breakpoint-md) { ... } /* Large devices (laptops and desktops up to 1199px) */ @media screen and (min-width: @breakpoint-lg) { ... } /* Extra large screens (screens up to 1600px) */ @media screen and (min-width: @breakpoint-xl) { ... } {% endhighlight %}

We occasionally expand on these media queries to include a max-width to limit CSS to a narrower set of devices.

{% highlight html %} @media (max-width: @breakpoint-sm-max) { ... } @media (min-width: @breakpoint-sm) and (max-width: @breakpoint-sm-max) { ... } @media (min-width: @breakpoint-md) and (max-width: @breakpoint-md-max) { ... } @media (min-width: @breakpoint-lg) { ... } {% endhighlight %}

Centered columns

Centered columns are placed in the middle of the row. This does not center their content, but centers the grid element, itself. This is accomplished by adding a class of .md-centered or .lg-centered depending on which breakpoint you want the column to center on. Items with the class .centered will carry through all breakpoints if not overridden by another version. You can center any number of columns you have. To have a column not carry it's centering to medium, add a class of .md-uncentered.

Building With Predefined HTML Classes

Block grids are made from a ul.sm-block-grid-# or ul.lg-block-grid-#. These are ideal for blocked-in content generated by an application, as they do not require rows or even numbers of elements to display correctly.

These have a bit of flexibility since you have access to two separate grids between our built-in 768px breakpoint. If you use the sm-block-grid only, the grid will keep its spacing and configuration no matter the screen size. If you use lg-block-grid only, the list items will stack on top of each other for small devices. If you use both of those classes combined, you can control the configuration and layout separately for each breakpoint.

Clearing Block Grid Rows

We have this built into the CSS, but you'll need to add a bit of code if you want your block grid to change layout between breakpoints. If you don't use this, it will inherit whatever you put in the smallest media query. This code looks for the n+1 of the grid so that it clears properly. That code looks like this:

{% highlight html %} .your-small-class-name > li { clear: none !important; } {% endhighlight %}

Note: If you are using the same class across breakpoints, your mixin should override this and you won't need it. If you want to control the layout using different classes, use the above snippet.

Built with Less

The typographic scale is based on two LESS variables in variables.less: @font-size-base and @line-height-base. The first is the base font-size used throughout and the second is the base line-height. We use those variables and some simple math to create the margins, paddings, and line-heights of all our type and more. Customize them and Strapit adapts.

Responsive utilities

For faster mobile-friendly development, use these utility classes for showing and hiding content by device via media query. Also included are utility classes for toggling content when printed.

Try to use these on a limited basis and avoid creating entirely different versions of the same site. Instead, use them to complement each device's presentation. Responsive utilities are currently only available for block and table toggling. Use with inline and table elements is currently not supported.
